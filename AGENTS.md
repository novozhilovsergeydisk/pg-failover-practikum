# Правила работы с кодовой базой

## Важно: Безопасность и контроль

1. **Не делать коммит, пуш и деплой без явной команды пользователя.** Всегда ждать подтверждения перед выполнением этих операций.

2. **Перед каждым коммитом выполнять анализ:**
   ```bash
   git status && git diff
   ```

3. **Проверять перед коммитом:**
   - Нет ли чувствительных данных (пароли, токены, API-ключи, .env файлы)
   - Нет ли потенциальных регрессий
   - Не изменилась ли логика существующего кода
   - Не изменились ли сигнатуры функций
   - Не нарушена ли обработка ошибок
   - Не изменились ли зависимости

4. **Выводить анализ на экран** в формате:
   - Что изменено
   - Потенциальные проблемы
   - Чувствительные данные: да/нет
   - Рекомендация: коммитить / не коммитить

5. **Если все в порядке** — делать деплой через скрипт `deploy` или `git-deploy` (после получения команды от пользователя).

6. **Формат коммитов** — Conventional Commits:
   - `feat:` — новая функциональность
   - `fix:` — исправление бага
   - `refactor:` — рефакторинг
   - `docs:` — документация
   - `style:` — стили
   - `test:` — тесты
   - `chore:` — техническая работа
